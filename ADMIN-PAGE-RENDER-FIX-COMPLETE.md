## 1. ROOT CAUSE

**STALE_SERVICE_WORKER**

The polished `/admin` UI was not rendering correctly in production because:

1. **Service Worker Precache**: The PWA service worker (`public/sw.js`) had a precache manifest with the old build ID (`2fbhhhei-jz5zOEFj0h8K`) and old admin page chunk (`app/admin/page-ae1d12296b8d49e1.js`)

2. **Aggressive Caching**: The service worker's `precacheAndRoute()` was serving the old cached version of the admin page chunk, even though the source code had been updated with Tailwind styling

3. **Cache Priority**: The precache manifest takes precedence over runtime caching rules, so even though `next.config.js` has a `NetworkOnly` rule for admin pages, the precached chunks were being served first

4. **User Impact**: Users with the old service worker installed continued to see the unstyled admin page because their browser was serving the cached JavaScript bundle that rendered the old UI

## TECHNICAL DETAILS

### Service Worker Lifecycle

1. **Detection**: Browser detects new `sw.js` file (byte-for-byte comparison)
2. **Installation**: New service worker installs in parallel with old one
3. **Waiting**: New service worker waits for old one to be released
4. **Activation**: `skipWaiting()` forces immediate activation
5. **Cleanup**: `cleanupOutdatedCaches()` removes old cached assets
6. **Control**: New service worker takes control of all pages

### Precache vs Runtime Caching

**Precache** (what was causing the issue):
- Assets listed in `precacheAndRoute()` are cached during service worker installation
- These are served directly from cache without network requests
- Takes precedence over runtime caching rules
- Requires service worker update to refresh

**Runtime Caching** (what we have configured):
- Rules defined in `next.config.js` for dynamic caching
- Admin pages set to `NetworkOnly` to avoid caching
- Only applies to assets NOT in the precache manifest

### Why Admin Pages Were Precached

Next.js PWA plugin automatically adds all static pages to the precache manifest, including admin pages. While we have a `NetworkOnly` runtime caching rule for admin navigations, the JavaScript chunks themselves were precached, causing the old UI to render.

## PREVENTION

### Future Considerations

1. **Exclude Admin from Precache** (optional):
   - Could configure PWA plugin to exclude admin chunks from precache
   - Would rely entirely on `NetworkOnly` runtime caching
   - Trade-off: Slower initial admin page load

2. **Version Monitoring**:
   - Monitor service worker version in production
   - Alert if users are stuck on old service workers
   - Implement forced update mechanism if needed

3. **Cache Busting**:
   - Next.js already does this with build IDs
   - Service worker updates automatically when `sw.js` changes
   - No additional action needed

### Current Configuration is Correct

The current PWA configuration in `next.config.js` is appropriate:
- `skipWaiting: true` ensures immediate activation
- `NetworkOnly` for admin navigations prevents caching admin HTML
- Precache for static assets improves performance
- The issue was simply that the service worker needed regeneration
