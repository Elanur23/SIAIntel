# Requirements Document

## Introduction

Controlled Remediation Phase 2B establishes a read-only Suggested Fixes Preview panel in the Warroom UI. This phase displays remediation suggestions generated by the Phase 2A generator to operators for review and manual consideration. The panel is strictly presentational with no mutation capabilities, no Apply to Draft functionality, and no automated fix application.

Phase 2B builds on the completed Phase 1 (types) and Phase 2A (generator) foundations to provide operators with visibility into potential remediation actions while maintaining fail-closed safety guarantees.

## Glossary

- **Suggested_Fixes_Preview_Panel**: A read-only UI component in Warroom that displays RemediationPlan and RemediationSuggestion data
- **RemediationPreviewPanel**: The React component implementing the preview panel
- **Advisory_Only**: Suggestions are informational and do not modify content or unlock gates
- **Preview_Mode**: Display-only mode with no mutation, no Apply button, no save behavior
- **Human_Review_Required**: Explicit acknowledgment that all suggestions require manual operator review and decision
- **Fail_Closed_UI**: UI design that preserves existing deploy gates and does not imply completion or safety

## Phase 1 and Phase 2A Foundation

**Phase 1 Deliverables (Completed and Deployed):**
- `lib/editorial/remediation-types.ts` - Type contract with RemediationSuggestion, RemediationPlan, enums, helpers
- `scripts/verify-remediation-engine.ts` - Type contract verification (8 tests)

**Phase 2A Deliverables (Completed and Deployed):**
- `lib/editorial/remediation-engine.ts` - Pure, deterministic, read-only generator
- `scripts/verify-remediation-generator.ts` - Generator verification (13 tests)

**Phase 2A Exports (Available for Phase 2B):**
- `generateRemediationPlan(input: GeneratorInput): RemediationPlan`
- `suggestionsFromGlobalAudit(globalAudit: unknown): RemediationSuggestion[]`
- `suggestionsFromPandaValidation(errors: unknown[]): RemediationSuggestion[]`
- `GeneratorInput` interface

**Phase 2A Safety Invariants (Must Be Preserved):**
- Generator is pure and deterministic
- No vault mutation, no article mutation
- cannotAutoPublish/Commit/Push/Deploy === true for all suggestions
- publishStillBlocked === true in all plans
- PROVENANCE_REVIEW, SOURCE_REVIEW, PARITY_REVIEW have suggestedText === null
- No source/provenance/E-E-A-T/fact/number fabrication

## Requirements

### Requirement 1: Read-Only Preview Panel

**User Story:** As a Warroom operator, I want to see suggested fixes for audit failures, so that I can understand potential remediation actions without any automated changes being applied.

#### Acceptance Criteria

1. THE Warroom UI SHALL provide a Suggested Fixes Preview panel
2. THE panel SHALL render a RemediationPlan summary showing total suggestions, critical count, human-only count
3. THE panel SHALL render individual RemediationSuggestion cards with metadata
4. THE panel SHALL be read-only (no mutation capabilities)
5. THE panel SHALL NOT have an Apply button
6. THE panel SHALL NOT have a Save button
7. THE panel SHALL NOT have a Publish button
8. THE panel SHALL NOT have any mutation behavior
9. THE panel SHALL NOT call API routes for persistence
10. THE panel SHALL NOT change deploy gate state
11. THE panel SHALL NOT modify vault content
12. THE panel SHALL NOT modify article content
13. THE panel SHALL display a zero-state message when no suggestions exist
14. THE panel SHALL be visually distinct from article content and deploy status indicators

### Requirement 2: Safe Placement in Warroom UI

**User Story:** As a Warroom operator, I want the preview panel placed clearly separate from article content, so that I don't confuse suggestions with actual article text or deploy readiness.

#### Acceptance Criteria

1. THE preview panel SHALL be placed as a separate CyberBox/card in the Warroom right panel
2. THE panel SHALL be positioned near or below the Global Health Board
3. THE panel SHALL NOT be embedded inside the article editor content area
4. THE panel SHALL NOT be placed where suggestions could be mistaken for current article text
5. THE panel SHALL default to collapsed or compact summary if the right panel is crowded
6. THE panel SHALL be visually separate from Deploy READY/PASS indicators
7. THE panel SHALL use distinct styling (border, background, header) to indicate preview-only status
8. THE panel SHALL have a clear header identifying it as "Suggested Fixes Preview" or equivalent
9. THE panel SHALL NOT obscure critical Warroom controls (Deploy button, Save button, audit results)
10. THE panel SHALL be scrollable independently if content exceeds available space

### Requirement 3: Remediation Plan Generation from Warroom State

**User Story:** As a developer, I want the preview panel to derive suggestions from existing Warroom state, so that no additional persistence or API calls are required.

#### Acceptance Criteria

1. THE UI SHALL use `generateRemediationPlan()` from Phase 2A to create the plan
2. THE UI SHALL pass `globalAudit` from Warroom state when available
3. THE UI SHALL pass `auditResult` if available and safe to use
4. THE UI SHALL pass Panda validation errors only if available in Warroom state
5. THE UI SHALL use `useMemo` or equivalent derived calculation to compute the plan
6. THE UI SHALL NOT persist remediation state to database
7. THE UI SHALL NOT persist remediation state to server
8. THE UI SHALL NOT write remediation state to vault
9. THE UI SHALL recompute the plan when input audit state changes
10. THE UI SHALL handle missing or undefined audit data gracefully (show zero-state)
11. THE UI SHALL NOT make network calls to fetch remediation data
12. THE UI SHALL derive the plan client-side from existing Warroom context

### Requirement 4: Preview-Only Warning Language

**User Story:** As a compliance officer, I want prominent warnings that suggestions are advisory only, so that operators understand no automated changes have been applied.

#### Acceptance Criteria

1. THE panel SHALL display a prominent warning banner stating "Preview only — not applied"
2. THE panel SHALL display a warning stating "Suggestions do not unlock Deploy"
3. THE panel SHALL display a warning stating "Human review required before any draft change"
4. THE panel SHALL display a warning stating "Re-run audit after any manual change"

6. THE warning SHALL remain visible when suggestions exist
7. THE warning SHALL use alert/warning styling (yellow/orange background or border)
8. THE warning SHALL be readable and not dismissible
9. HUMAN_ONLY suggestions SHALL clearly explain why no suggested text is generated
10. THE panel SHALL NOT use language implying suggestions have been applied or verified

#### Acceptance Criteria

2. THE UI SHALL NOT use the word "Corrected"

## Safety Reminders

**CRITICAL SAFETY RULES:**

1. **Preview Only:** Panel is read-only, no Apply button, no mutation
2. **Fail-Closed UI:** Existing deploy gates preserved, no unlock behavior
3. **Advisory Language:** Use "Suggested", "Preview", "Proposed" - never "Fixed", "Applied", "Verified"
4. **No Fabrication:** Do not generate sources, provenance, E-E-A-T, facts, or numbers in UI
5. **Human-Only Categories:** SOURCE_REVIEW, PROVENANCE_REVIEW, PARITY_REVIEW clearly indicate human verification required
6. **Warning Banners:** Prominent warnings that suggestions are advisory only and do not unlock Deploy
7. **No Side Effects:** Component is purely presentational, no API calls, no database writes, no vault mutation
8. **Minimal Page Changes:** One import, one JSX insertion in page.tsx - no deploy/audit logic changes
9. **No Regression:** All existing tests pass, no existing functionality broken
10. **File Discipline:** Only 2 files changed (RemediationPreviewPanel.tsx created, page.tsx modified)

**If any of these rules are violated, Phase 2B is BLOCKED.**
