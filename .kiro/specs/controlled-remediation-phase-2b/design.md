## 1. Overview
Controlled Remediation Phase 2B introduces a read-only **Suggested Fixes Preview** panel into the Warroom. This panel serves as an advisory interface, displaying suggestions generated by the Phase 2A remediation engine without applying any changes to the article draft or the vault.

### 1.1 Core Principles
- **Preview Only**: Strictly presentational. No automated or semi-automated application of fixes.
- **Read-Only**: No state persistence, no API writes, no article mutations.
- **Fail-Closed**: Existing deploy gates and audit requirements remain unchanged and fully enforced.
- **Advisory Language**: Clear distinction between "Current State" and "Suggested Preview".

---

## 2. Architecture

### 2.1 Component Structure
- **RemediationPreviewPanel.tsx**: A pure, stateless (UI-only state excepted) React component that handles the rendering of suggestions.
- **page.tsx**: Minimal integration point. One import and one JSX insertion in the right-panel sidebar.

### 2.2 Data Flow
1. **Source State**: `page.tsx` provides `globalAudit`, `auditResult`, `pandaValidationErrors`, and `deployLockReasons` via props.
2. **On-the-fly Generation**: `RemediationPreviewPanel` internally calls `generateRemediationPlan` from `lib/editorial/remediation-engine.ts`.
3. **Memoization**: The plan is computed using `useMemo` with audit data and IDs as dependencies.
4. **No Persistence**: Suggestions are never saved to the database, localStorage, or server-side cache.

### 2.3 System Constraints
- **No API Routes**: No new backend endpoints.
- **No Vault Mutation**: The `vault` state in `page.tsx` is never modified by this component.
- **No Side Effects**: No `useEffect` hooks that trigger external actions or re-audits.

## 4. Derived State Strategy

### 4.1 Remediation Plan Computation
The component will use `useMemo` to compute the `RemediationPlan` whenever the input props change.

```typescript
const remediationPlan = useMemo(() => {
  return generateRemediationPlan({
    articleId,
    packageId,
    globalAudit,
    pandaValidationErrors,
    deployLockReasons,
    auditResult
  });
}, [articleId, packageId, globalAudit, pandaValidationErrors, deployLockReasons, auditResult]);
```

### 4.2 UI State
- Local `useState` allowed ONLY for:
  - `isExpanded`: boolean (to toggle the entire panel)
  - `expandedSuggestions`: Record<string, boolean> (to toggle individual suggestion cards)

## 5. UI Placement

### 5.1 Right Panel Sidebar
The panel will be placed within the Warroom right-panel sidebar, utilizing the existing `CyberBox` pattern.

- **Position**: Below the **Global Health Board** section and above/near the **Deploy Lock Reasons** section.
- **Visual Distinction**:
  - Header: "Suggested Fixes Preview"
  - Accent Color: Advisory/Warning (e.g., Yellow/Gold `[#FFB800]`)
  - Initial State: Collapsed or Compact if the sidebar is crowded.

---

## 6. Panel Structure

### 6.1 Header
- Title: `SUGGESTED FIXES PREVIEW`
- Badge: `READ-ONLY ADVISORY` (Gray/Dimmed)

### 6.2 Warning Banner (Mandatory)
A permanent, non-dismissible alert at the top of the panel:
> [!WARNING]
> **PREVIEW ONLY — NOT APPLIED**
> Suggestions do not unlock Deploy. Human review required before any draft change.

### 6.3 Summary Stats
- **Total Suggestions**: Total count
- **Critical**: Red badge for critical severity
- **Requires Approval**: Yellow badge for human-review-gate
- **Human-Only**: Gray badge for system-refusal (e.g., SOURCE_REVIEW)

### 6.4 Suggestion List
- Rendered as a list of `SuggestionCard` components.
- Zero-state: "No suggested fixes available for current audit state."

### 6.5 Footer
- Message: "Re-run audit after any manual change."

## 7. Suggestion Card Design

### 7.1 Visual Layout
Each card must be visually distinct from the article editor and the audit results.

- **Header Section**:
  - Category Badge (e.g., `FORMAT_REPAIR`)
  - Severity Badge (Color-coded: Red/Orange/Yellow/Blue)
  - Safety Level Badge (e.g., `REQUIRES_HUMAN_APPROVAL`)
- **Body Section**:
  - **Issue**: Short description of the problem.
  - **Rationale**: Why this fix is proposed.
  - **Suggested Change**:
    - If `suggestedText` is present: Displayed in a read-only code/pre block.
    - If `suggestedText` is null: Display "Human review required — no automated suggestion."
- **Metadata**:
  - `Language`: e.g., "EN"
  - `Field`: e.g., "body"
- **Footer Section**:
  - "Does not unlock Deploy." (Small text)

---

## 8. Human-Only Handling
The design explicitly prevents the generation of fabricated data for high-risk categories.

- **SOURCE_REVIEW**:
  - Message: "Evidence required — human must verify and add source attribution."
- **PROVENANCE_REVIEW**:
  - Message: "Provenance required — human must verify provenance data."
- **PARITY_REVIEW**:
  - Message: "Human truth-source required — numeric/entity verification needed."
- **Styling**: Human-only cards should have a distinct, more neutral background (e.g., `bg-white/5`) compared to safely-suggested cards.

---

## 9. Safe Wording Rules

### 9.1 Allowed Wording
- `Suggested`, `Preview`, `Advisory`, `Proposed`, `Potential`, `Human review required`, `Recommended`.

### 9.2 Forbidden Wording (Strict Enforcement)
- `Fixed`, `Corrected`, `Applied`, `Auto-applied`, `Ready for Publish`, `SIA Protocol Satisfied`, `Verified Fix`, `Safe to Deploy`, `Source Added`, `Provenance Verified`.

---

## 10. Side-Effect Restrictions
The `RemediationPreviewPanel` component **must not**:
- Call `fetch()` or any API wrapper.
- Call `vault[lang].setter` or any article modification function.
- provide a Copy-to-Clipboard button (Phase 3 feature).
- provide an Apply-to-Draft button (Phase 3 feature).
- Persist state to `localStorage`.

---

## 11. Gate Preservation
- **isDeployBlocked**: Logic in `page.tsx` must remain untouched.
- **isPublishing**: Behavior must remain untouched.
- **Audit Triggers**: Buttons for "Run Global Audit" must not be modified or auto-triggered.
- **Deploy Readiness**: No logic shall allow the preview of a fix to bypass a `LOCKED` state.

---

## 12. Styling and Accessibility
- **Compactness**: Use `text-[10px]` to `text-xs` for metadata.
- **Headings**: Semantic `h3` or `h4` for panel sections.
- **ARIA**: `aria-expanded` for collapsible sections.
- **Contrast**: High contrast for badges (White text on color background).
- **Keyboard**: Expand/Collapse must be triggerable via Enter/Space.
